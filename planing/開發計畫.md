# TraceRedis 開發計畫 (Development Roadmap)

基於《技術文件》的內容，TraceRedis 是一個具備「執行追蹤 (Execution Tracing)」能力的記憶體資料庫 (Redis-like)。
以下為整體的開發 Roadmap，預計開發時間為 6~8 週，共包含 5 個主要的里程碑 (Milestones)。強烈建議每完成一個里程碑，便在 GitHub 上建立對應的 Release 版本 (如 v0.1, v0.2等)，以保持專案進度的明確性與專業度。

---

## Milestone 1: TCP Server Core (v0.1)
**目標**：建立可以接受 Client 連線的非同步 TCP 伺服器

- **Task 1: 專案初始化**
  - 建立專案目錄：`traceredis/`
  - 建立模組目錄：`network/`, `protocol/`, `commands/`, `storage/`, `trace/`, `cli/`, `web/`, `tests/`
  - 提交第一個 Commit：`project skeleton`
- **Task 2: TCP Server 基礎**
  - 使用 Python `asyncio.start_server` 建立非同步伺服器。
  - 實作 `handle_client(reader, writer)` 以接受連線並印出請求內容。
- **Task 3: Connection Manager**
  - 為每個 Client 連線建立 Read Loop (`while True: read command`)。
- **Task 4: Basic Response**
  - 實作基礎的回覆機制，例如回傳 RESP 協定的 `+OK\r\n`。

---

## Milestone 2: Redis Protocol (v0.2)
**目標**：讓伺服器可以解析並理解 Redis Command (對應 RESP 協定)

- **Task 5: RESP Parser**
  - 實作 RESP 協定解析器 (例如將 `*2\r\n$3\r\nGET\r\n$3\r\nkey\r\n` 解析出來)。
- **Task 6: Command Object**
  - 建立 `Command` 類別封裝解析結果 (包含 `name` 與 `args` 屬性)。
- **Task 7: Command Dispatcher**
  - 建立 Command Routing 機制 (`command_table` 字典對應各指令與其處理函數)。
- **Task 8: Basic Commands**
  - 實作最基礎的測試指令以驗證流程：`PING` (回覆 `PONG`) 與 `ECHO`。

---

## Milestone 3: In-Memory Database (v0.3)
**目標**：實作完整的 Key-Value Store 與核心指令邏輯

- **Task 9: Storage Engine**
  - 建立基礎的記憶體儲存結構 (簡單的 `store = {}` dict)。
  - 實作 `Entry` 類別來存放值 (包含 `value` 與 `expire_at` 屬性)。
- **Task 10: SET 指令**
  - 實作 `SET key value` (流程：Lookup -> Store -> Return OK)。
- **Task 11: GET 指令**
  - 實作 `GET key` 以取得對應的 Value。
- **Task 12: DEL 指令**
  - 實作 `DEL key` 刪除功能。
- **Task 13: TTL System (過期機制)**
  - 實作 `EXPIRE key seconds`。
  - 第一版架構採用 Lazy Expiration：僅在呼叫 `GET` 等讀取指令時，才檢查 key 是否過期並清除。

---

## Milestone 4: Trace Engine (v0.4) - ✨專案靈魂核心
**目標**：讓每個指令的執行步驟都被詳細記錄，落實「Everything is traceable」，此為 TraceRedis 的最大特色。

- **Task 14: TraceSession**
  - 實作 `TraceSession` 類別，代表單次指令的完整生命週期與多筆事件 (Events) 集合。
- **Task 15: TraceEvent**
  - 實作 `TraceEvent` 類別，記錄單獨的執行步驟 (名稱、時間戳記、Metadata)。
- **Task 16: TraceCollector**
  - 實作追蹤資料管理中心，建議使用 `collections.deque(maxlen=1000)` 作為 Memory Ring Buffer，保留最近的 1000 筆紀錄避免記憶體溢出。
- **Task 17: Trace API 與 Context**
  - 提供開發者所使用的 Tracer API：`tracer.start()`, `tracer.event()`, `tracer.end()`。
  - 使用 Python `contextvars` 儲存當前 async 行程的 current trace context。
- **Task 18: Command Tracing 整合**
  - 於各指令的執行流程中埋點 (Emit event)。
  - 例如：`parse_command`, `dispatch_command`, `hash_lookup`, `write_value`, `ttl_register`, `response_send`。

---

## Milestone 5: Visualization (v1.0)
**目標**：將收集到的 Trace 執行資料透過 Web 介面視覺化呈現。

- **Task 19: Trace REST API**
  - 實作與其對應的端點 (如 `GET /traces`, `GET /trace/{id}`)，輸出 JSON 格式的追蹤資料 (不記錄耗時，只記 Timestamp 由前端計算 Delta)。
- **Task 20: Web Dashboard**
  - 第一個版本能顯示 Command List 與具體的 Trace Timeline (Waterfall Chart) 供使用者觀察。
  - 使用前端框架 (例如 React) 打造，呈現出類似 Chrome DevTools 的 Network Waterfall 視覺效果。

---

## 統整架構與模組分層設計

- `network/`: 處理由 TCP 連線與讀寫，不牽涉邏輯處理。
- `protocol/`: 解析 RESP 協定將其轉換成內部使用之 `Command` Object。
- `commands/`: 負責 Command 路由分派。
- `engine/`: 掌管核心指令的執行邏輯。
- `storage/`: 負責 In-memory dict 與 Lazy Expiration 的 TTL 管理。
- **`trace/`**: 專案靈魂所在；包含 Session、Event、Collector 與 ContextVars 狀態管理。
- `web/` / `cli/`: Web 儀表板的 REST API 以及未來提供給使用者操作的可追蹤 CLI 客戶端 (`traceredis-cli`)。
